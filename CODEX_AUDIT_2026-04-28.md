## Codex Audit Report

### Code Audit Findings

- `src/roles/writer.ts:145-150` lets the writer choose arbitrary output paths and writes `join(root, c.file)` without proving the final path stays inside `root`. A model response containing `../...` can overwrite files outside the requested review target. This is the highest-risk issue because `fix` mode intentionally writes to disk.

- `action.yml:12-19` exposes `mode` and `max-cost-usd`, but `src/cli.ts:286-290` only maps `INPUT_MODE=fix` to `--autofix`, `src/cli.ts:161-162` accepts `--autofix`, and the `pr` handler still always calls `runReviewMode` at `src/cli.ts:210-215`. The action's `fix` mode and `max-cost-usd` input are effectively no-ops.

- SAST file paths are not normalized before aggregation or PR filtering. `src/sast/eslint.ts:35` stores ESLint `filePath` as returned by ESLint, `src/sast/semgrep.ts:54` stores Semgrep `path`, while AI findings use `readSourceTree` relative paths from `src/util/files.ts:90`. Aggregation keys and `postPrReview` path lookups are exact-string based (`src/findings/aggregate.ts:42-45`, `src/reporters/github-pr.ts:50-55`), so SAST findings can fail to merge with AI findings and can be dropped from PR comments.

- `src/commands/init.ts:247-249` accepts `max_iterations: 0`, and the generated config comment at `src/commands/init.ts:375-379` says zero is valid, but the schema rejects it with `z.number().int().min(1)` at `src/config/schema.ts:39`. A user can generate or test a config that `loadConfig` will not accept.

- `review.parallel: false` is not a true sequential mode. `src/modes/review.ts:89-100` constructs all reviewer promises before deciding whether to `Promise.all` or await them sequentially, so calls start concurrently either way. This matters for users trying to avoid provider rate limits.

- Gate limits are evaluated only inside the fix loop at `src/modes/fix.ts:228-245`. Initial scan cost (`src/modes/fix.ts:142-143`) and final verification cost (`src/modes/fix.ts:264-301`) can exceed `max_cost_usd` or wall-time limits without setting `gateBlocked`, especially when the loop is skipped.

- PR file listing is not paginated. `src/cli.ts:194-199` requests `per_page: 300` once. PRs with more than 300 files can have missing commentable-line maps, causing findings in later files to be treated as untouched and omitted.

- `scope` is defined in config (`src/config/schema.ts:17-18`) but is never applied in review or fix mode. Reviewers receive the full `files` array in `src/modes/review.ts:89-93` and `src/modes/fix.ts:117-123`, so a configured reviewer scope has no effect.

- A single malformed model finding can discard an entire reviewer run. `parseFindings` parses the full payload through Zod at `src/findings/parse.ts:109-132`, and `runReviewer` turns any parse error into zero findings at `src/roles/reviewer.ts:86-105`. Per-finding salvage would be more robust.

- `output.report`, `output.findings`, and `output.diff` are schema fields (`src/config/schema.ts:50-54`) but the CLI writes hard-coded `review-<timestamp>` and `fix-<timestamp>` paths under `--output-dir` (`src/cli.ts:89-99`, `src/cli.ts:132-142`). Users configuring `output:` get ignored values.
